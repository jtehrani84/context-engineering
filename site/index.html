The Problem

Without Context Engineering

  • Every session starts from zero
  • Same corrections repeated weekly
  • AI-sounding output requires manual editing
  • Knowledge trapped in chat history
  • No immune response to mistake patterns

With Context Engineering

  • Every session picks up where you left off
  • Corrections stick permanently (rules + hooks)
  • Voice quality enforced mechanically
  • Knowledge compounds across sessions
  • The system gets better without effort

11 Layers, One System

Each layer reinforces the others. Memory feeds rules. Rules feed hooks. Hooks feed skills. Skills feed the graph.

1

Identity (CLAUDE.md)

Who you are, what you do, routing table for context. Loaded every session.

2

Rules

Voice standards, security, architecture. Soft enforcement via context.

3

Hooks

Hard enforcement. Code that runs mechanically. Can't be forgotten.

4

Skills (21)

Workflow automation. /research-prep, /draft, /validate. 30 min -> 2 min.

5

Memory

Corrections, decisions, preferences. Every "remember this" persists.

6

Wiki

Structured knowledge. People, entities, concepts, projects. Graph-indexed.

7

Knowledge Graph

SQLite graph that grows from your work. Surfaces related context automatically.

8

Passive Intelligence

Overnight crons gather web + HN + GitHub intel while you sleep.

9

Multi-Model QA

Cross-validate content against multiple models. /validate before shipping.

10

Meta-Skills

/skillify extracts new skills from sessions. The system builds itself.

11

Compound Loop

Mistake -> correction -> memory -> rule -> hook -> prevention. Permanent immunity.

The Compound Loop

Day 1:  You correct Claude → Memory file saved

Day 3:  Same mistake class → Rule prevents it

Day 7:  Rule forgotten in long session → Hook enforces

Day 14: Hook catches pattern → /skillify extracts it

Day 30: Crons feed intel → Morning brief surfaces it

Day 60: Graph connects entities → Better context always

A mediocre setup running for 60 days beats a perfect setup running for 1 day.
